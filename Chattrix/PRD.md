Project Overview Chattrix is a full-featured real-time chat application modeled after WhatsApp. It supports text and emoji-only communication, user profiles, story-style statuses, and read receipts. It is built to run on web browsers and as a native Android/iOS app using Capacitor.   1.1 Architecture Overview Layer Technology / Description Frontend React 18 + Vite 5 (TypeScript recommended) Mobile Wrapper Capacitor 5 — wraps React web app for Android & iOS Backend / Database Supabase (PostgreSQL, Realtime, Auth, Storage) Authentication Supabase Auth — Email/Password + Google OAuth 2.0 File Storage Supabase Storage — avatars bucket Realtime Supabase Realtime — WebSocket subscriptions for messages & status State Management React Context API + useReducer (or Zustand for scale) Routing React Router v6 Styling Tailwind CSS + custom CSS variables Build Tool Vite 5 with SWC plugin Package Manager pnpm (preferred) or npm   1.2 Key Constraints •       Android: App must NOT run fullscreen — preserve the native status bar and navigation buttons. In Capacitor config, set backgroundColor and ensure no fullscreen plugin is used. •       Media: No photos, videos, or audio. Only text messages and emoji characters. •       Status: Text + emojis only. Auto-expires in 24 hours. •       Auth: Users must be authenticated to access any feature. •       Row-Level Security: All Supabase tables must use RLS policies to isolate user data.   2. Supabase Backend — Full Specification 3.1 Environment Setup •       Create a new project at https://supabase.com •       Copy Project URL and anon public key into .env file as VITE_SUPABASE_URL and VITE_SUPABASE_ANON_KEY •       Enable Google OAuth in Authentication > Providers > Google (requires Google Cloud Console credentials) •       Enable Email/Password sign-in in Authentication > Providers > Email •       In Authentication > URL Config, set Site URL to your domain (or http://localhost:5173 for dev)   3.2 Database Schema Table: users Extends Supabase's built-in auth.users table via a trigger. Stores public profile data. Column Type Notes id uuid Primary key — references auth.users(id) ON DELETE CASCADE email text Unique, not null — synced from auth.users phone text Nullable — entered by user on profile name text Nullable initially, set on first login or profile edit dob date Nullable — user's date of birth dp_url text Nullable — public URL to avatar in Supabase Storage last_login timestamptz Updated on every auth event via trigger created_at timestamptz DEFAULT now() is_online boolean DEFAULT false — updated by presence system   SQL Migration — 001_users.sql: CREATE TABLE public.users (   id         uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,   email      text UNIQUE NOT NULL,   phone      text,   name       text,   dob        date,   dp_url     text,   is_online  boolean DEFAULT false,   last_login timestamptz,   created_at timestamptz DEFAULT now() );   -- Trigger: auto-insert into users when a new auth user is created CREATE OR REPLACE FUNCTION public.handle_new_user() RETURNS trigger AS $$ BEGIN   INSERT INTO public.users (id, email, created_at)   VALUES (NEW.id, NEW.email, now())   ON CONFLICT (id) DO NOTHING;   RETURN NEW; END; $$ LANGUAGE plpgsql SECURITY DEFINER;   CREATE TRIGGER on_auth_user_created   AFTER INSERT ON auth.users   FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();   Table: chats Represents a 1-to-1 conversation between two users. No group chats. Column Type Notes id uuid PRIMARY KEY DEFAULT gen_random_uuid() user1_id uuid REFERENCES public.users(id) ON DELETE CASCADE user2_id uuid REFERENCES public.users(id) ON DELETE CASCADE last_message_id uuid Nullable — FK to messages(id), updated after every send created_at timestamptz DEFAULT now()   CREATE TABLE public.chats (   id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),   user1_id        uuid NOT NULL REFERENCES public.users(id) ON DELETE CASCADE,   user2_id        uuid NOT NULL REFERENCES public.users(id) ON DELETE CASCADE,   last_message_id uuid,   created_at      timestamptz DEFAULT now(),   UNIQUE (user1_id, user2_id) );   -- Ensure user1_id < user2_id to prevent duplicate pairs CREATE OR REPLACE FUNCTION normalize_chat_users() RETURNS trigger AS $$ BEGIN   IF NEW.user1_id > NEW.user2_id THEN     NEW.user1_id := OLD.user2_id; NEW.user2_id := OLD.user1_id;   END IF;   RETURN NEW; END; $$ LANGUAGE plpgsql;   CREATE TRIGGER normalize_chat_before_insert   BEFORE INSERT ON public.chats   FOR EACH ROW EXECUTE FUNCTION normalize_chat_users();   Table: messages Column Type Notes id uuid PRIMARY KEY DEFAULT gen_random_uuid() chat_id uuid REFERENCES public.chats(id) ON DELETE CASCADE sender_id uuid REFERENCES public.users(id) ON DELETE CASCADE content text The message text or emoji string — max 4000 chars type text CHECK (type IN ('text', 'emoji')) DEFAULT 'text' status text CHECK (status IN ('sent', 'seen')) DEFAULT 'sent' deleted_at timestamptz Nullable — soft delete timestamp created_at timestamptz DEFAULT now()   CREATE TABLE public.messages (   id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),   chat_id    uuid NOT NULL REFERENCES public.chats(id) ON DELETE CASCADE,   sender_id  uuid NOT NULL REFERENCES public.users(id) ON DELETE CASCADE,   content    text NOT NULL CHECK (char_length(content) <= 4000),   type       text NOT NULL DEFAULT 'text'              CHECK (type IN ('text', 'emoji')),   status     text NOT NULL DEFAULT 'sent'              CHECK (status IN ('sent', 'seen')),   deleted_at timestamptz,   created_at timestamptz DEFAULT now() );   CREATE INDEX idx_messages_chat_id ON public.messages(chat_id); CREATE INDEX idx_messages_created_at ON public.messages(created_at DESC);   Table: status Column Type Notes id uuid PRIMARY KEY DEFAULT gen_random_uuid() user_id uuid REFERENCES public.users(id) ON DELETE CASCADE content text Text + emoji — max 700 characters background_color text Hex color string e.g. '#FFD700' — default is white created_at timestamptz DEFAULT now() expires_at timestamptz DEFAULT now() + interval '24 hours'   CREATE TABLE public.status (   id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),   user_id          uuid NOT NULL REFERENCES public.users(id) ON DELETE CASCADE,   content          text NOT NULL CHECK (char_length(content) <= 700),   background_color text NOT NULL DEFAULT '#FFFFFF',   created_at       timestamptz DEFAULT now(),   expires_at       timestamptz DEFAULT now() + interval '24 hours' );   CREATE INDEX idx_status_user_id ON public.status(user_id); CREATE INDEX idx_status_expires_at ON public.status(expires_at);   3.3 Row-Level Security Policies RLS must be enabled on ALL tables. Run ALTER TABLE <table> ENABLE ROW LEVEL SECURITY; for each.   Migration — 005_rls_policies.sql: -- USERS TABLE ALTER TABLE public.users ENABLE ROW LEVEL SECURITY; CREATE POLICY "Users can view all profiles"   ON public.users FOR SELECT USING (true); CREATE POLICY "Users can update own profile"   ON public.users FOR UPDATE USING (auth.uid() = id);   -- CHATS TABLE ALTER TABLE public.chats ENABLE ROW LEVEL SECURITY; CREATE POLICY "Users see own chats"   ON public.chats FOR SELECT   USING (auth.uid() = user1_id OR auth.uid() = user2_id); CREATE POLICY "Users create chats they belong to"   ON public.chats FOR INSERT   WITH CHECK (auth.uid() = user1_id OR auth.uid() = user2_id);   -- MESSAGES TABLE ALTER TABLE public.messages ENABLE ROW LEVEL SECURITY; CREATE POLICY "Users see messages in their chats"   ON public.messages FOR SELECT   USING (chat_id IN (     SELECT id FROM public.chats     WHERE user1_id = auth.uid() OR user2_id = auth.uid()   )); CREATE POLICY "Users send messages in their chats"   ON public.messages FOR INSERT   WITH CHECK (     sender_id = auth.uid() AND     chat_id IN (SELECT id FROM public.chats       WHERE user1_id = auth.uid() OR user2_id = auth.uid())   ); CREATE POLICY "Users update own message status"   ON public.messages FOR UPDATE   USING (sender_id = auth.uid() OR chat_id IN (     SELECT id FROM public.chats     WHERE user1_id = auth.uid() OR user2_id = auth.uid()   ));   -- STATUS TABLE ALTER TABLE public.status ENABLE ROW LEVEL SECURITY; CREATE POLICY "Users see all active statuses"   ON public.status FOR SELECT   USING (expires_at > now()); CREATE POLICY "Users create own status"   ON public.status FOR INSERT   WITH CHECK (auth.uid() = user_id); CREATE POLICY "Users delete own status"   ON public.status FOR DELETE   USING (auth.uid() = user_id);   3.4 Supabase Storage — Avatars Bucket •       Bucket name: avatars •       Visibility: Public (so dp_url can be loaded without auth headers) •       File naming convention: {user_id}.jpg or {user_id}.png •       Max upload size: 2MB — enforce client-side before upload •       Allowed MIME types: image/jpeg, image/png, image/webp   -- Storage Policy: Only authenticated owner can upload CREATE POLICY "Allow user to upload their own avatar"   ON storage.objects FOR INSERT   WITH CHECK (     bucket_id = 'avatars' AND     auth.uid()::text = (storage.foldername(name))[1]   ); CREATE POLICY "Allow user to update their own avatar"   ON storage.objects FOR UPDATE   USING (     bucket_id = 'avatars' AND     auth.uid()::text = (storage.foldername(name))[1]   ); CREATE POLICY "Public read access to avatars"   ON storage.objects FOR SELECT   USING (bucket_id = 'avatars');   3.5 Supabase Realtime Channels Channel / Event Purpose messages:INSERT on messages (filtered by chat_id) Push new messages instantly to both users in the chat messages:UPDATE on messages (filtered by chat_id) Update read receipt (sent → seen) — triggers blue ticks status:INSERT on status (no filter) Push new statuses to all users' status feeds presence channel per chat_id Track who is typing and online/offline per conversation chats:INSERT / UPDATE (filtered by user) Update chat list order when new message arrives   Important: Supabase Realtime must be enabled per table in the Supabase dashboard under Database > Replication. Enable INSERT, UPDATE, DELETE for messages, status, and chats tables.   4. Notifications & Chat Ordering Logic 10.1 In-App Notification Logic 1.    User A sends message to User B 2.
